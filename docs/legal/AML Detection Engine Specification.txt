AurumShield
AML Detection Engine Specification

Entity: PVN LLC d/b/a AurumShield
Document Type: Technical Compliance Specification
Purpose: Automated financial crime detection

1. Monitoring Architecture

The AurumShield AML monitoring engine operates using a multi-layered detection architecture.

The system evaluates:

Customer Risk
+
Transaction Risk
+
Counterparty Risk
+
Jurisdiction Risk
+
Blockchain Exposure Risk
=
Composite Risk Score

Transactions exceeding defined thresholds trigger compliance alerts.

2. Transaction Risk Scoring Model

Each transaction is assigned a risk score between 0–100.

Score Range	Risk Classification
0–30	Low Risk
31–60	Medium Risk
61–80	High Risk
81–100	Critical Risk

Actions triggered:

Risk Score	Action
<60	Transaction proceeds
60–79	Compliance alert
80–89	Transaction paused
90+	Automatic freeze
3. Monitoring Thresholds
High-Value Transaction Monitoring

Because AurumShield processes large B2B settlements, baseline thresholds are elevated.

Alert triggers:

Rule	Threshold
Single transaction size	> $250,000
Daily volume spike	> 300% baseline
Weekly turnover	> $5M
4. Velocity Monitoring

The system detects abnormal transaction velocity.

Triggers include:

• More than 5 settlements within 1 hour
• > $1M cumulative settlement within 24 hours for new accounts
• Settlement bursts inconsistent with customer profile

5. Structuring Detection

Structuring detection monitors attempts to avoid reporting thresholds.

Triggers include:

• Multiple transfers between $9,000–$10,000
• Repeated settlements just below internal monitoring limits
• Coordinated structuring across related entities

Structuring alerts automatically escalate to compliance.

6. Counterparty Risk Monitoring

Counterparty analysis evaluates relationships between entities.

Triggers include:

• Rapid asset transfers between newly created entities
• Circular settlement patterns
• Settlement chains with no commercial purpose

7. Geographic Risk Detection

High-risk jurisdiction monitoring includes:

• FATF grey list jurisdictions
• FATF black list jurisdictions
• Sanctioned regions

Transactions involving these jurisdictions automatically generate alerts.

8. Blockchain Risk Monitoring

Stablecoin transactions are evaluated using wallet risk scoring.

Indicators include exposure to:

• sanctioned wallets
• ransomware wallets
• darknet marketplaces

High-risk wallet exposure results in transaction rejection.

9. Sanctions Screening

Sanctions screening occurs:

During onboarding

Prior to settlement

During periodic re-screening

Data sources include:

• OFAC SDN List
• EU Consolidated Sanctions List
• UN Sanctions List

Positive matches trigger automatic account freeze.

10. Behavioral Risk Monitoring

Customer activity is evaluated against historical patterns.

Triggers include:

• sudden change in settlement size
• rapid increase in transaction frequency
• settlement behavior inconsistent with business activity

11. Machine-Readable Rule Engine

The AML monitoring system enforces rules defined in a structured format.

Example rule definition:

RULE: HIGH_VALUE_SETTLEMENT

IF
transaction_amount > 250000
AND
customer_account_age < 30 days

THEN
risk_score += 40
trigger_alert = TRUE
12. Alert Severity Levels

Alerts generated by the system are categorized as:

Level	Description
Level 1	Informational
Level 2	Compliance review required
Level 3	Settlement paused
Level 4	Account freeze
13. Investigation Workflow

When an alert is generated:

Step 1 — Compliance analyst review
Step 2 — Customer documentation review
Step 3 — Enhanced due diligence
Step 4 — SAR determination

14. SAR Decision Tree

SAR filing is required when:

• suspicious activity exceeds $2,000
• funds appear linked to illicit activity
• structuring is detected
• sanctions evasion is suspected

SARs are filed via the FinCEN BSA E-Filing System.

15. Audit Logging

All AML system decisions are recorded in immutable logs.

Logged data includes:

• transaction metadata
• triggered rules
• risk score calculations
• compliance decisions

Logs are stored for 5 years.

16. Model Governance

The monitoring model is reviewed annually.

Reviews evaluate:

• rule effectiveness
• false positive rates
• regulatory updates

17. Compliance Overrides

Only the Compliance Officer may override system decisions.

All overrides require documented justification.

18. Fail-Closed Settlement Control

The settlement engine integrates directly with the AML monitoring system.

Settlement flow:

Transaction Initiated
↓
AML Risk Evaluation
↓
Risk Score Calculation
↓
Pass → Settlement Executes
Fail → Transaction Halted

No settlement may occur without AML clearance.
